BOTGI GST BILLING IMPLEMENTATION PLAN
Date: 2026-04-01
Project: botgi_project Django web app

1. DECISION FIRST

Yes, this can be implemented successfully in the current web app.

Recommended direction:
- Continue with the existing Django web app.
- Do not rebuild the whole system as a separate PC app now.
- If needed later, package the same web app as a Windows desktop wrapper only after the GST workflow is stable.

Why the current web app is the right base:
- The project already has inventory, party master, product master, invoice numbering, billing screens, print views, roles, and admin controls.
- The missing work is GST compliance logic and document structure, not a platform problem.
- A web app is easier to update when GST rules change.
- Multi-user staff access is easier in a web app than in a single-machine desktop build.

When a PC app makes sense:
- You need full offline-first billing on one machine.
- You need deep printer, barcode scanner, local device, or local database integration.
- You want a single-shop installation with no browser/server workflow.

Current recommendation:
- Build GST billing in this Django app.
- Improve the backend architecture and database before calling it production-ready.


2. CURRENT CODEBASE ASSESSMENT

Existing reusable foundation already present:
- Product master
- Inventory party master
- Inventory bill header and line entries
- Sales invoice number sequencing
- Job billing workflow
- Product sale ledger
- Company profile with GST toggle and default GST rate
- Inventory print views and staff permission system

Main gaps for full GST billing:
- No proper GST master data on product and party records
- No canonical GST document layer for mixed service + product billing
- No CGST/SGST/IGST split engine based on place of supply
- No HSN/SAC, UQC, tax category, cess, reverse charge support
- No GSTR-1 table mapping
- No GSTR-2B / IMS / purchase reconciliation flow
- No filing-period lock, amendment tracking, or audit-level return workflow
- Current database is SQLite, which is not ideal for serious concurrent billing usage

Important architecture note:
- Inventory stock movement and GST return documents are related, but they are not the same thing.
- The current app stores stock movement well, but GST needs a stronger invoice/document model on top of that.


3. MUST-DO TECHNICAL CHANGES BEFORE FULL ROLLOUT

3.1 Database
- Move from SQLite to PostgreSQL or MySQL before production GST billing rollout.
- Reason: concurrent edits, locking, reporting load, and financial reliability.

3.2 Code Organization
- Move tax calculation and GST validation into dedicated helper/service modules.
- Do not leave tax logic spread across templates and views only.

3.3 Testing
- Add model, service, and workflow tests for:
  - GST calculations
  - invoice numbering
  - inter-state vs intra-state tax split
  - credit/debit note reversal
  - stock reversal
  - GSTR summaries

3.4 Audit and Locking
- Add filing period lock.
- Add audit log for invoice changes, cancellations, notes, GST edits, and return regeneration.


4. TARGET IMPLEMENTATION ARCHITECTURE

Recommended approach:
- Keep current inventory models for stock ledger operations.
- Add a proper GST document layer for compliance-ready billing.

Suggested model direction:
- Keep using Product, InventoryParty, InventoryBill, InventoryEntry for stock and operational register use.
- Introduce a new canonical sales/purchase tax document structure for GST compliance.

Suggested new document models:
- GstDocument
- GstDocumentLine
- GstReturnPeriod
- GstReturnSnapshot
- GstReconciliationImport
- GstReconciliationLine
- CreditDebitNote
- EInvoiceLog

Why a separate GST document layer is safer:
- Current job billing mixes service lines and product sale lines.
- InventoryEntry is good for stock movement but not ideal as the only tax document source.
- A GST document layer can represent:
  - service-only invoices
  - product-only invoices
  - mixed job invoices with service + spare parts
  - credit notes and debit notes
  - exports, SEZ, reverse charge, exempt lines

6. BEST IMPLEMENTATION ORDER FOR THIS PROJECT

Recommended order:
1. Phase 1 - GST master data foundation
2. Phase 2 - GST calculation engine
3. Phase 3 - GST document engine
4. Phase 4 - purchase and ITC support
5. Phase 5 - GSTR-1 and GSTR-3B reports
6. Phase 6 - 2B and IMS reconciliation
7. Phase 8 - lock and audit controls
8. Phase 7 - e-invoice and e-way bill

Reason for this order:
- Master data and tax engine are prerequisites for everything else.
- Document structure must be correct before reports and returns are trusted.
- Reconciliation and e-invoice come after the core invoice workflow is stable.


7. WHAT WE SHOULD START WITH NOW

Start with PHASE 1 - GST MASTER DATA FOUNDATION.

Why this is the best first slice:
- Lowest risk
- High future value
- Does not break current billing flow badly
- Unlocks tax engine and GST reporting work

8. RISKS TO WATCH

- SQLite concurrency issues if multiple users edit billing together
- Mixing stock ledger and tax document logic too tightly
- Keeping tax math in JavaScript only
- Missing audit trail for edited invoices
- Trying to build GSTR exports before master data and tax engine are stable
